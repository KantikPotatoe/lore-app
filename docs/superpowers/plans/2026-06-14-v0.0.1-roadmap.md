# Roadmap to v0.0.1

**Date:** 2026-06-14
**Scope:** First real release — **solo use only** (just me). Multi-user, easy
distribution, and packaging are explicitly out of scope and deferred to a later
version.

## Framing

The app is already feature-rich for a "0.0.1": lore pages, typed infoboxes,
DB-backed page types/templates, `[[wiki links]]`, maps with pins, the
relationship graph + hubs/orphans, a customizable home, and JSON backup/import.

So 0.0.1 is **not** about adding features. It's about being able to **trust the
app with real worldbuilding data**. Two things can cause unrecoverable pain:
losing the database, and silently breaking the link graph. Everything in Tier 1
targets those. Features are Tier 2. Anything about *how others run it* is Tier 3
and deliberately postponed.

The release itself is small: finish Tier 1, optionally pull in one Tier 2 item,
bump `package.json` `version` from `0.0.0` → `0.0.1`.

---

## Tier 1 — Trust (must-have before relying on it)

### 1.1 Durable, low-effort backups

**Problem.** All data lives in one browser's IndexedDB at `localhost:5174`. The
only safety net is a manual JSON export the user must remember to run. IndexedDB
can be evicted; a cleared profile or origin change shows an empty DB. This is the
single highest-risk area.

**Do.**
- Use the **File System Access API** to keep a persistent folder handle and write
  a timestamped JSON automatically — on a schedule and/or after every N edits —
  with no save dialog after the first grant. This converts "hope the user
  exports" into "it backs itself up."
- Fallback for when the API/permission isn't available: keep the existing
  `downloadBackup()` path.
- Surface last-backup time and "N edits since backup" prominently on Home and in
  the banner.

**Files.** `src/backup.ts` (new folder-handle logic + scheduler),
`src/components/BackupBanner.tsx`, `src/routes/HomeRoute.tsx`, possibly a new
`meta` row for the stored directory handle / cadence.

**Cross-ref.** See `2026-06-14-stable-storage-launcher.md` for prior
storage-hardening work; reconcile with `requestPersistentStorage()` already
called on app start.

**Done when.** After a one-time folder grant, edits produce timestamped backup
files on disk automatically, and Home clearly shows backup freshness.

### 1.2 Import safety (no silent wipe)

**Problem.** `importAll()` **replaces** the entire database — no merge, no undo.
A mis-click on Import after the on-disk file has diverged silently destroys
current data.

**Do.**
- Confirmation dialog that states the consequence in concrete numbers:
  "Replace current N pages with M pages from this file. This cannot be undone."
- Auto-export the current DB to a `pre-import` backup file immediately before
  applying an import.

**Files.** `src/db.ts` (`importAll`), the Home import UI in
`src/routes/HomeRoute.tsx`.

**Done when.** Importing always shows counts + requires confirmation, and a
recovery backup of the pre-import state is written first.

### 1.3 Link integrity on rename and delete

**Problem (confirmed in code).** Links resolve by title at render time
(`getOrCreatePageByTitle`, `db.ts:518`; backlinks/graph match lowercased titles).
- **Rename:** title changes go through `updatePage` (`db.ts:506`) with no link
  rewrite, so every existing `[[Old Title]]` reference breaks. Clicking a broken
  link then **creates a new empty stub** — actively corrupting the wiki.
- **Delete:** `deletePage` (`db.ts:510`) just removes the row. Inbound links
  become dangling and re-create stubs when clicked; backlinks silently vanish.

**Do.**
- On rename, rewrite references across all pages: body `<a data-wikilink>` anchors
  (`data-title` + visible text) **and** infobox `[[…]]` / `ref` field values.
  Implement as a `renamePage(id, newTitle)` helper rather than burying it in
  `updatePage`.
- On delete, decide and implement a policy: either leave links as visibly "broken"
  (and stop auto-creating a stub on click for a known-deleted title), or rewrite
  them to plain text. Recommended: mark broken so the user notices, and make
  clicking a broken link **not** silently spawn a stub.
- Consider a small "broken links" report (reuse the hubs/orphans machinery in
  `GraphRoute` / `HubsOrphansPanel`).

**Files.** `src/db.ts` (new `renamePage`, delete policy, a link-rewrite util
shared with `linkedTitles`), `src/routes/PageRoute.tsx` (call rename helper),
`src/extensions/WikiLink.ts` / `LoreEditor.tsx` (broken-link click behavior).

**Done when.** Renaming a referenced page keeps all links working; deleting a
page no longer lets clicks resurrect it as an empty stub.

## Tier 2 — High-value features (pull in if cheap; else 0.0.2)

### 2.1 Inline images in article bodies *(recommended for 0.0.1 if time allows)*

**Problem.** The editor is bare `StarterKit` (`LoreEditor.tsx:36`) — images only
exist in infoboxes and maps. A worldbuilding wiki really wants portraits/scenes
inside prose.

**Do.** Add the Tiptap image extension; store as data URLs to stay local-first;
add a size guard / downscale (reuse `imageUtils.ts`) so the DB doesn't bloat.

**Files.** `src/components/LoreEditor.tsx`, `src/imageUtils.ts`, `src/index.css`.

### 2.2 External links

**Problem.** You can make `[[wiki links]]` but not link out to a reference URL —
`StarterKit` omits the Link extension.

**Do.** Add the Tiptap Link extension with a toolbar button; open external links
in a new tab; keep `WikiLink` handling separate.

**Files.** `src/components/LoreEditor.tsx`.

### 2.3 Tables in the editor

Lower priority. Useful for stat blocks / timelines. Tiptap table extension.
Defer unless trivially added alongside 2.1/2.2.

## Tier 3 — Deferred (explicitly NOT in 0.0.1)

These matter only when others run it; postponed per "make the app easier to run
much later."

- **Distribution / packaging.** Today the launcher runs the Vite **dev** server.
  Eventually: `vite preview` against a built `dist/`, then a PWA
  (`vite-plugin-pwa`) for an installable offline app. Not now.
- **LICENSE file.** None at root. Decide before sharing publicly. Not needed for
  solo use.
- **Automated tests.** None today (by design). Revisit once the data model and
  link-integrity rules stabilize.
- **Empty / first-run state polish.** What a brand-new zero-page DB looks like.
